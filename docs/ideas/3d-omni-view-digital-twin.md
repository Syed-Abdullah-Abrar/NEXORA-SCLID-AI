# 3D Omni-View Digital Twin

## Problem Statement
How might we build a synchronized, real-time 3D simulation of a hypothetical disaster that provides tailored, performant visual intelligence to everyone from EOC commanders down to street-level civilians without buckling under heavy API integrations during a hackathon?

## Recommended Direction
**The "Synchronized Sandbox" (All-in-One 3D Demo).**
Instead of a live physics engine processing real-world APIs, we will build a **pre-scripted, high-fidelity 3D WebGL/Three.js environment** of a hypothetical city block. The core innovation is that the *same* 3D timeline is synchronized across all three user roles via WebSockets, but their *camera perspective and data overlays* change based on their role:
1.  **EOC Commander (Top-Down):** A macro-view of the 3D city. Features a "Time-Travel Slider" to fast-forward the flood simulation and see predictive risk overlays generated by the Early Warning Agent.
2.  **Field Marshall (Voxel/Fog of War):** A tactical view where the city is obscured. As simulated HAM radio bursts arrive, the "fog" clears and 3D damage voxels light up in the exact GPS coordinates of the report.
3.  **Affected Civilian (Street-Level AR):** A ground-level camera perspective. They watch the 3D floodwaters rise relative to their position, with a glowing, AR-style safe route projected onto the 3D street by the Resource Allocation Agent.

## Key Assumptions to Validate
- [ ] **Assumption 1 (Performance):** Three.js can smoothly render a city block model and water particle effects simultaneously in a browser. *Test:* Load a free GLTF city model and 10,000 particles immediately to check FPS drops.
- [ ] **Assumption 2 (Synchronization):** We can sync a master "disaster timestamp" across three browser windows using a simple WebSocket server so all roles see the flood happening at the exact same moment. *Test:* Build a barebones Socket.io server broadcasting a `tick` event.
- [ ] **Assumption 3 (UI Decoupling):** The Figma engineer can design DOM-based UI overlays (glass-morphism panels, task graphs) that float *above* the WebGL canvas without z-index or pointer-event conflicts. *Test:* Float a heavy HTML `div` over a basic Three.js scene.

## MVP Scope
**IN:**
- A single, static 3D GLTF city model (downloaded, not custom modeled).
- A basic rising plane (blue) to simulate floodwaters.
- A global "Disaster Timeline" slider (0 to 100%) that controls the water height.
- Three distinct camera angles (Macro, Mid-range, Street) mapped to the 3 roles.
- Hardcoded agent overlays (Task Graph, Risk Heatmap) that appear at specific timestamps.

**OUT:**
- Real-world map generation (Mapbox/Google Earth 3D tiles). Too heavy for a hackathon.
- Real-time physics or fluid dynamics.
- Actual LLM generation during the 3D render (use pre-generated JSON responses to guarantee smooth 60 FPS playback).

## Not Doing (and Why)
- **Live API Integration:** Calling real weather or traffic APIs during a 3D render introduces latency and unpredictable hackathon failures. Pre-script the data.
- **Complex 3D Animations:** No people running or cars driving. It distracts from the core value proposition: the *Agents orchestrating the response*, not the visual effects of the disaster itself.
- **Heavy React/Three.js Frameworks (React Three Fiber):** If the engineer is not an expert, stick to Vanilla Three.js. R3F can introduce complex lifecycle bugs that kill hackathon momentum.
